# Requirements - Tegne

This file defines the user-visible product contract. Keep implementation rules in `.claude/rules/` and formal diagram syntax in `.claude/specs/`.

## Purpose

Tegne is a browser-based structural and visual modelling tool. Users write plain-text DSL files, render them as interactive SVG diagrams, adjust layout by dragging, then save or export the result.

Tegne is not a simulator. It does not evaluate equations, run models, or execute system behaviour.

## Supported Diagrams

| `@type` | Diagram | Extension | Specification |
|---|---|---|---|
| absent or `sd` | Forrester stock-and-flow | `.sd` | `.claude/specs/stock-flow.md` |
| `id` | Integration diagram | `.id` | `.claude/specs/integration-diagram.md` |
| `infoflow` | Information flow diagram | `.iff` | `.claude/specs/information-flow-diagram.md` |
| `tm` | Threat model | `.tm` | `.claude/specs/threat-model.md` |

Unknown `@type` values must produce a parse error.

## Core User Workflow

1. Write or open a DSL file.
2. Render the diagram.
3. Fix parse errors while the last valid diagram remains visible.
4. Drag elements, groups, and annotation boxes to refine layout.
5. Save the DSL with `@position` directives preserving layout.
6. Export a standalone SVG for documentation or presentations.

## Shared DSL Directives

All diagram types support these metadata directives unless the diagram spec states otherwise:

```text
@type         sd|id|infoflow|tm
@name         <model name>
@version      <version string>
@date         <date string>
@author       <author name>
@theme        dark|light|tokyo
@orientation  landscape|portrait
@size         a4|a3|a2|a1|a0
@show-ids     on|off
@legend       on|off
@info         on|off
```

Defaults:

- missing `@type` means `sd`
- missing `@date` becomes today's ISO date
- missing `@theme` means `dark`
- missing `@orientation` means `landscape`
- missing `@size` means `a4`
- missing `@show-ids` means `off`
- missing `@legend` means `on` where the diagram has a legend
- missing `@info` means `on` (metadata info box rendered)

All diagram types support same-type shared setup files:

```text
@include <filename>
```

Requirements:

- Includes resolve against files loaded by Config, Open Folder, or an equivalent project load.
- Single-file open does not resolve sibling files; unresolved includes must produce a clear parse error.
- Included files must be the same diagram type as the host.
- Included files may contribute dictionaries, display defaults, and threat-model `@ref` entries only.
- Included identity metadata, positions, diagram elements, and nested includes must not affect the host.

## Layout Persistence

`@position` directives are generated by Save and restore manual layout:

```text
@position <id> <x> <y>
```

Reserved position IDs:

| ID | Meaning |
|---|---|
| `__meta__` | metadata box |
| `__legend__` | legend box |
| `__mitigations__` | threat-model mitigations panel |
| `__stride_key__` | threat-model STRIDE key |

Requirements:

- Save must emit one `@position` line for every draggable diagram element.
- If some positions are missing, positioned elements keep their saved coordinates and the rest fall back to auto-layout.
- Position coordinates are numeric and are rounded when written.

## User Interface

The application uses a two-column layout:

- left column: toolbar, DSL editor, error panel
- right column: canvas toolbar, SVG canvas, Help panel

UI requirements:

- Open accepts `.sd`, `.id`, `.iff`, and `.tm`.
- Render parses the current editor text and redraws the diagram.
- Parse errors appear below the editor with line numbers when available.
- A parse error must not clear the last valid diagram.
- Save writes DSL text plus current positions using the browser File System Access API when available, with download fallback.
- Export SVG produces a standalone SVG at full page zoom and strips internal `data-*` attributes.
- Current editor text persists in `localStorage`.
- The Help panel must describe the implemented DSL syntax for every diagram type.

## Canvas Behaviour

- The SVG viewport represents the selected paper size and orientation.
- Canvas background and diagram colours come from the active theme.
- Zoom controls support zoom in, zoom out, and reset.
- Mouse wheel or trackpad pans the canvas.
- Zoom resets to 100% on each render.
- Draggable elements update connected lines, groups, and affected annotations immediately.

## Themes

Supported themes:

| Theme | Requirement |
|---|---|
| `dark` | default muted dark theme |
| `light` | light background theme |
| `tokyo` | vivid dark theme with glow where supported |

All semantic colours must be defined in `src/themes.ts`. Renderers must not introduce new hardcoded semantic colours.

## Documentation Requirements

When behaviour, syntax, rendering, or workflow changes, update all affected files in the same change:

- `.claude/requirements.md`
- relevant `.claude/specs/*.md`
- relevant `.claude/rules/*.md`
- `README.md`
- `index.html` Help panel
- `fixtures/`
- tests

## Acceptance Requirements

A change is complete only when:

- implementation matches this file and the relevant diagram spec
- parser tests cover valid and invalid syntax
- fixture examples cover realistic usage
- renderer or geometry tests cover shape, edge, legend, or state visual changes
- focused tests pass
- full test suite and production build are run for shared or browser-facing changes when feasible
